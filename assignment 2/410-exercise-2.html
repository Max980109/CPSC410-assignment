Procedures are the New Variables!

In class you have learned how to extend a simple DSL with support for variables: When variable declarations or assignments get evaluated, the variable names and and values get stored in a symbol table. When a variable is accessed, its current can then be looked up in that symbol table.

While variables are probably the most commonly found “real language feature” in DSLs, there is another one that we will investigate in this exercise: Procedures!

When you think of procedures, you might intuitively think about functions, or methods in object-oriented languages. For now we will consider a form of procedures that are much simpler: We assume that procedures do not take any parameters or return anything. Speaking in Java-terms, they need to be equivalent to static void Proc() { ... }.

It might surprise you how similar implementing support for procedures is to implementing support for variables: The core idea is once again to store the procedure name and its value (here: the procedure body) in a symbol table and to look it up when the procedure is called.

Exercise 2:

Implement support for simple procedures (without parameters or return values), starting from the code for TinyVars from class.

The grammar for the extended language is:

PROGRAM     := PROCDEF* STATEMENT*
PROCDEF     := "def" NAME PROCBODY
PROCBODY    := "{" STATEMENT* "}"
STATEMENT   := DEC | SET | PRINT | PROCCALL
DEC         := "new" NAME
SET         := "set" NAME "," EXP
PRINT       := "print" EXP 
PROCCALL    := "call" NAME
EXP         := NAME | NUMBER
NAME        := a-z+
NUMBER      := 0-9+

An example program that you should support is:

def swap {
    new temp
    set temp, a
    set a, b
    set b, temp
}

new a
set a, 1
new b
set b, 2
call swap
print a
print b
call swap 
print a
print b
  1. Add the new literals to the tokenizer. Note that { and } are reserved characters in regular expressions so you might have to escape them using \\{ and \\}.
  2. Add AST nodes for the new rules PROCDEF, PROCBODY and PROCCALL. You need to allow PROCBODYs to be stored into the symbol table. You could do so either by changing the value type of the table EXP and make PROCBODY extend EXP, or you could make the value type of the symbol table more generic, e.g. Object. Either way you might need to type-cast then reading from the symbol table.
  3. Modify the parse() method for the PROGRAM AST node , and implement the method for the new nodes.
  4. Modify the evaluate() method for the PROGRAM AST node, and implement the method for the new nodes.
    • When a PROCDEF node is evaluated, the NAME of the procedure is stored in the symbol table with the PROCBODY as its expression value. The PROCBODY itself should not be evaluated at this point!
    • When a PROCCALL node is evaluated, it looks up the procedure name in the symbol table and evaluates the PROCBODY stored in there.
    • When a PROCBODY is evaluated, it simply evaluates all of its statements in order, like the PROGRAM node.

We strongly recommend that you test your implementation thoroughly before you continue!

Besides all of the design decisions discussed in Exercise 3, we made another major decision: Our procedures cannot take parameters or return anything. As even the simple example program in Exercise 2 shows, this makes programs using them somewhat “ugly” and limits their applicability in practice. It would be much nicer if we could have real functions in our DSLs. Fortunately, we will see that creating real functions doesn’t require too much extra work.

Turning Procedures into Functions

Let us start by extending our previous version of procedures so that they can return an expression value:

Exercise 4:

Extend your code from exercise Exercise 2 with support for return values.

The grammar for the extended language is almost identical to the previous one, with two minor modifications:

...
PROCBODY    := "{" STATEMENT* ("return" EXP)? "}"
...
EXP       := NAME | CONST | PROCCALL
...

An example program that you should support is:

def swap {
    set a, call innerswap
}

def innerswap {
    new temp
    set temp, b
    set b, a
    return temp
}

new a
set a, 1
new b
set b, 2
call swap
print a
print b
call swap
print a
print b
  1. Add the new "return" literal to the Tokenizer.
  2. Modify the parser and evaluator for PROCBODY to allow an optional return value between the last STATEMENT and the } token. The returned expression should be evaluated after all the statements and the result should be returned by the PROCBODY.
  3. Modify PROCCALL to be useable as an expression. Unfortunately, Java does not allow PROCCALL to be both a STATEMENT and an EXP. You could solve this by converting STATEMENT into an interface, or by creating a second class (e.g. PROCCALLEXP) that is a copy of PROCCALL but extends EXP instead of STATEMENT.
  4. Don’t forget to update the makeExp method in EXP.

Now that we have return values, we can finally update our procedures one last time to make them take parameters and turn them into “real” functions:

Exercise 5:

Extend your code from exercise Exercise 4 with support for parameters.

As for the previous extension, you only need to make a few localized modifications:

...
PROCBODY    := ( "(" NAME ("," NAME)* ")" )? "{" STATEMENT* "}"
...
PROCCALL    := "call" NAME ( "(" EXP ("," EXP)* ")" )?
...

An example program that you should support is:

def swap(x, y) {
    set a, y
    set b, x
}

new a
set a, 1
new b
set b, 2
call swap(a, b)
print a
print b
call swap(b, a)
print a
print b
  1. Add the new ( and ) literals to the Tokenizer. You might have to escape them since they are reserved regular expression characters.
  2. Modify the parser for PROCBODY and PROCCALL to allow optional parameters for function definitions and calls. Remember that you also need to modify PROCCALLVAL if you decided to duplicate PROCCALL in the previous exercise.
  3. Modify the evaluator for PROCBODY and PROCCALL. PROCCALL needs to evaluate the expression for each argument and then put the result into the symbol table, very similar to variable assignments. Implementing this can however be a bit tricky since the information is spread out across multiple AST nodes. One way of getting this to work is to add a field in PROCBODY that contains the arguments and that gets set by PROCCALL. When PROCBODY is evaluated, it then has all the necessary information to prepare the symbol table before evaluating its statements.

As you are done with Exercise 5, your code now supports full procedures that can both take parameters and return values. While this is a big step up compared to the original TinyVars language, you might feel that something is still odd. In particular, when you look at all the example programs that we gave you during the exercises, they are no behaving as you would expect from a normal “program”.

The reason why all of these programs seem odd is that our procedures do not have their own scope. This means is that they can freely access any variables defined around them, and also define their own variables that surrounding code can then access. Therefore, while we might have support for parameters and return values, neither are actually necessary.

Look at the sample program in Exercise 5: Instead of returning a or b, we just set them directly and these changes carry over into the surrounding code. Even worse: If you look at the symbol table print-out at the end of evaluating the program you see that even the parameters x and y are exposed to the surrounding program.

Let us go back to our code one last time to fix this behaviour:

Exercise 6:

Extend your code from exercise Exercise 5 with support for scoping.

No changes to the grammar are necessary for this, since all we are modifying are the semantics of our evaluator.

To support scoping, we need to give each function its own local symbol table instead using the same global one. Each AST node further needs to know which symbol table to refer to when it is evaluated.

  1. Add a new symbol table field to PROCBODY.
  2. Change the signature of the evaluate method to add a symbol table parameter.
  3. Update all AST nodes that refer to the symbol table to use the one they received as a parameter instead of the global one.
  4. Update PROCBODY/PROCCALL to store function arguments into the PROCBODY‘s symbol table, and evaluate statements and the return value using this table as well.

With this last set of changes, we can now support programs like this one and don’t need to worry anymore about accidentally overwriting variables inside functions:

def swap(a, b) {
    new temp
    set temp, a
    set a, b
    set b, temp
    return a
}

new a
set a, 1
new b
set b, 2
new swapped
set swapped, call swap(a, b)
set b, call swap(b, a)
set a, swapped
print a
print b

As you may notice, the downside of this change is however that we also cannot support a proper swapping function anymore since we can only return one expression value at a time.

Some languages solve this problem by allowing functions to directly overwrite the variables they’ve received as parameters. This way, functions have a “channel” to the outside program that is less restricted than in our language but still easier to control than what we had without scoping.

The following example program that shows how simple our swapping function could be implemented in this “call-by-reference” style:

def swap(ref a, ref b) {
    new temp
    set temp, a
    set a, b
    set b, temp
}

new a
set a, 1
new b
set b, 2
call swap(a, b)
print a
print b

Unfortunately, supporting call-by-reference requires a memory model, similar to the one you’ve seen in class to support aliasing. While extending TinyVars with this feature might be a useful exercise, we will skip it for now and only consider what call-by-reference support means in theory.
